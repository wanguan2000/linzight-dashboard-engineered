# LinZight GA Release Checklist

Use this checklist before publishing a GA functional-testing branch, tag, or release artifact.

Current candidate: `v1.0.2`.

## Required Local Gates

Run these from the repository root:

```bash
npm run lint
npm run build
npm run smoke:api
npm run smoke:crf-semantics
npm run export:openapi
npm run export:html
npm run smoke:ui
npm run smoke:static-runtime
npm run smoke:performance
npm run regression:browser
npm run browser:matrix
npm run demo:e2e
npm run export:postgres-migration -- exports/postgres-migration
npm run backup:postgres
npm run deploy:staging
npm run release:check
npm run smoke:docker
npm test
docker compose config
python3 -m compileall -q backend
git diff --check
git status --short --branch
```

## Version And Release Notes

- Confirm `package.json` version matches the planned GA tag.
- Update `CHANGELOG.md` before creating a tag or GitHub release.
- Suggested GA tag format: `v1.0.2`.
- Release notes should call out that this is a GA functional-testing release for demo/test data entry, not a real-patient production clinical deployment.
- Rollback path for the static frontend is the previous `dist/` or `exports/html/` artifact; rollback path for the backend is the previous source tag plus a verified PostgreSQL backup and upload/object-storage backup.

## What `npm run release:check` Verifies

- Required package scripts are present.
- Core handoff, changelog, API, protocol, OpenAPI snapshot, release-readiness, production RC workflow, permission matrix, UAT package, frontend audit, deployment operations, and HTML export docs exist.
- GitHub Actions runs lint, build, backend compile, API smoke, CRF semantics smoke, OpenAPI export, HTML export, UI smoke, static runtime smoke, browser matrix, demo E2E, performance smoke, the release gate, and Docker smoke.
- Browser regression is layered via `npm run regression:browser`; if Playwright is not installed, the script writes a limitation report under `reports/`.
- Role/Study browser matrix is layered via `npm run browser:matrix`; if Playwright is not installed, the script writes a skipped report under `reports/browser-matrix.json`.
- End-to-end customer demo chain is layered via `npm run demo:e2e`; it validates `admin@demo.linzight`, `lung-crc@demo.linzight`, and `lung-dm@demo.linzight` from login through patient queue, CRF, eConsent, sample/testing, Journey, analytics, Query, Approval Center, and backend operation-log coverage.
- PostgreSQL formal migration scripts exist under `backend/migrations/postgres/`; `npm run export:postgres-migration` remains a legacy SQLite test-data export package for staging rehearsal.
- Staging deploy plan is generated by `npm run deploy:staging` and records validation plus rollback steps.
- OpenAPI export script and `docs/openapi.json` exist.
- Docker smoke script exists and validates compose build/up, backend health/login, and frontend app shell.
- If the formal PostgreSQL database already has an initial admin whose password was changed after bootstrap, Docker smoke validates the existing GA bootstrap state through the backend container database connection instead of requiring the default initial password.
- If the Docker registry is temporarily unavailable but the project images already exist locally, `npm run smoke:docker` can fall back to cached `linzight-dashboard-engineered-backend/frontend:latest` images after the build timeout; fresh CI or a new machine without cached images still fails as expected.
- Docker Compose baseline files exist for local frontend/backend PostgreSQL startup.
- Legacy SQLite backup/restore scripts and deployment operations notes exist.
- Static export manifest has all eight module pages.
- Tracked files do not include `.env`, local databases, upload payloads, dependency folders, build caches, private keys, or SQLite files.
- Large tracked files outside `resource/` are blocked.

## Manual Release Notes

- `backend/linzight_demo.db`, `.env*`, upload payloads, `node_modules/`, `.next/`, `dist/`, and caches must stay untracked.
- `exports/html/` is generated by `npm run export:html`; regenerate it after frontend changes.
- The GA functional-testing runtime uses PostgreSQL and still uses local HMAC-signed Bearer tokens. Real patient deployments still need managed secrets, TLS, backup retention, centralized identity, production object storage, and PostgreSQL RLS policies.
- Formal PostgreSQL runtime must keep `/seed` disabled unless a controlled reset window explicitly sets `LINZIGHT_ENABLE_SEED_ENDPOINT=1` and uses an LZ Admin token.
- System Management `Operation Logs` must read from backend `operation_logs` through Study-scoped APIs and export CSV; the removed Audit Diff module must stay removed.
- `backups/` is ignored and is only for legacy local SQLite/upload backups.

## Formal Product Release Blockers

- Confirm every cross-Study admin table displays `Study ID` and every create/export/quality action writes an explicit `study_id`.
- Keep expanding the role x Study x endpoint permission matrix beyond the current smoke coverage, especially around real identity provider groups and multi-center assignments.
- Before handling real patient data, replace the local PostgreSQL functional-testing deployment with managed PostgreSQL, validated indexes, RLS policies, constraints and backup/restore drills; SQLite is not an allowed formal runtime.
- Continue hardening Query management, backend operation-log retrieval/export, export download traceability, eConsent withdraw/re-sign and visit-window alerts with reviewer UX, reporting and supplier-level validation. Data lock and de-identification approval remain internal governance workflows outside this customer-facing release scope.
- Verify mobile table/card layouts and browser matrix screenshots for all eight module pages before a customer-facing production release.
